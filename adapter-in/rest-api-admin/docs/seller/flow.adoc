[[seller-flow]]
= 셀러 입점 및 운영 플로우

이 문서는 셀러 입점 신청부터 승인, 운영까지의 전체 플로우를 설명합니다.

== Phase 1: 입점 신청

=== 1.1 입점 신청 제출

신청자가 입점에 필요한 정보를 제출합니다.

**API**: `POST /api/v2/admin/seller-applications/apply`

**필수 정보**:

* 셀러 기본 정보 (셀러명, 표시명, 로고, 설명)
* 사업자 정보 (사업자등록번호, 회사명, 대표자명, 통신판매업 신고번호)
* 주소 정보 (반품 주소)
* CS 연락처 정보

**신청 상태**: `PENDING` (대기)

=== 1.2 신청 시 검증

* 필수 필드 입력 확인
* 사업자등록번호 형식 검증 (10자리 숫자)
* 연락처 형식 검증

== Phase 2: 신청 심사

=== 2.1 신청 목록 조회

관리자가 대기 중인 신청 목록을 조회합니다.

**API**: `GET /api/v2/admin/seller-applications`

**필터 옵션**:

* 신청 상태 (PENDING, APPROVED, REJECTED)
* 검색어 (셀러명, 사업자등록번호)
* 페이지네이션

=== 2.2 신청 상세 확인

특정 신청의 상세 정보를 확인합니다.

**확인 항목**:

* 제출된 사업자 정보의 정확성
* 연락처 정보의 유효성
* 기타 신청 조건 충족 여부

== Phase 3: 승인/거절 처리

=== 3.1 승인 처리

**API**: `POST /api/v2/admin/seller-applications/{id}/approve`

**승인 시 수행되는 작업**:

[source]
----
1. 검증 단계
   ├── 셀러명 중복 확인
   └── 사업자등록번호 중복 확인

2. Seller 생성 단계
   ├── Seller (기본 정보) 생성
   ├── SellerAddress (반품 주소) 생성
   ├── SellerBusinessInfo (사업자 정보) 생성
   └── SellerCs (CS 정보) 생성

3. 상태 업데이트
   ├── 신청 상태 → APPROVED
   └── 승인된 Seller ID 저장

4. 이벤트 발행
   └── SellerApplicationApprovedEvent
----

**응답**: 생성된 Seller ID 반환

=== 3.2 거절 처리

**API**: `POST /api/v2/admin/seller-applications/{id}/reject`

**요청 필드**:

* `rejectReason`: 거절 사유 (필수)
* `processedBy`: 처리자 정보

**거절 시 수행되는 작업**:

* 신청 상태 → `REJECTED`
* 거절 사유 저장
* 처리자 정보 저장

== Phase 4: 셀러 운영

승인된 셀러에 대해 다양한 관리 작업을 수행합니다.

=== 4.1 셀러 조회

**관리자용 상세 조회**: `GET /api/v2/admin/sellers/{sellerId}`

* 기본 정보, 주소, 사업자 정보, CS, 계약, 정산 정보 포함

**목록 검색**: `GET /api/v2/admin/sellers`

* 활성화 여부, 검색어, 정렬, 페이지네이션 지원

=== 4.2 셀러 정보 수정

**기본 정보 수정**: `PUT /api/v2/admin/sellers/{sellerId}`

* 셀러명, 표시명, 로고, 설명 수정

**전체 정보 수정**: `PUT /api/v2/admin/sellers/{sellerId}` (확장 요청)

* 기본 정보 + 주소 + CS + 사업자 정보 일괄 수정

=== 4.3 추가 정보 관리

승인 후 관리자가 추가로 설정하는 정보:

* **계약 정보**: 수수료율, 계약 기간, 특별 조건
* **정산 정보**: 정산 계좌, 정산 주기

== API 호출 예시 시나리오

=== 시나리오 1: 신규 셀러 입점 승인

[source,bash]
----
# 1. 입점 신청
POST /api/v2/admin/seller-applications/apply
{
  "sellerInfo": { "sellerName": "테스트셀러", ... },
  "businessInfo": { "registrationNumber": "123-45-67890", ... },
  "addressInfo": { ... }
}
# Response: { "applicationId": 1 }

# 2. 관리자: 신청 목록 조회
GET /api/v2/admin/seller-applications?status=PENDING

# 3. 관리자: 승인 처리
POST /api/v2/admin/seller-applications/1/approve
{
  "processedBy": "admin@example.com"
}
# Response: { "sellerId": 100 }

# 4. 생성된 셀러 조회
GET /api/v2/admin/sellers/100
----

=== 시나리오 2: 입점 신청 거절

[source,bash]
----
# 1. 관리자: 거절 처리
POST /api/v2/admin/seller-applications/2/reject
{
  "rejectReason": "사업자등록증 확인 불가",
  "processedBy": "admin@example.com"
}
----
