## Redis 캐싱 전략

| 항목 | 값 |
|------|-----|
| Redis Key 패턴 | `seller::{sellerId}` |
| 캐시 전략 | Cache-Aside (Look-Aside) |
| TTL | 없음 (명시적 만료 미설정) |
| 직렬화 방식 | JSON (JsonUtils.toJson / JsonUtils.fromJson) |
| Redis 값 타입 | String (StringRedisTemplate) |

주의: `SellerRedisQueryServiceImpl.saveSellerInRedis()`가 `save(key, value)` 오버로드를 사용하여 TTL 없이 저장한다. `RedisKey.SELLER`에 1시간 TTL이 정의되어 있으나(`hour = 1`), 실제 저장 시 `save(key, value, duration)` 대신 `save(key, value)`가 호출되므로 TTL이 적용되지 않는다. 셀러 정보 변경 시 캐시가 자동으로 무효화되지 않는다.

### QueryDSL 코드

```java
// SellerFindRepositoryImpl.java - fetchSeller()
queryFactory
    .select(
        new QSellerInfo(
            seller.id,
            sellerBusinessInfo.companyName,
            seller.sellerLogoUrl,
            seller.sellerDescription.coalesce(""),
            sellerBusinessInfo.businessAddressLine1
                .concat(" ")
                .append(sellerBusinessInfo.businessAddressLine2)
                .concat(" ")
                .append(sellerBusinessInfo.businessAddressZipCode),
            sellerBusinessInfo.csNumber.coalesce(sellerBusinessInfo.csPhoneNumber),
            sellerBusinessInfo.csPhoneNumber,
            sellerBusinessInfo.registrationNumber,
            sellerBusinessInfo.saleReportNumber,
            sellerBusinessInfo.representative,
            sellerBusinessInfo.csEmail))
    .from(seller)
    .innerJoin(sellerBusinessInfo)
    .on(sellerBusinessInfo.id.eq(seller.id))
    .where(seller.id.eq(sellerId))
    .fetchFirst()
```

### 동등 SQL

```sql
SELECT
    s.seller_id,
    sbi.company_name,
    s.seller_logo_url,
    COALESCE(s.seller_description, '') AS seller_description,
    CONCAT(sbi.business_address_line1, ' ',
           sbi.business_address_line2, ' ',
           sbi.business_address_zip_code) AS address,
    COALESCE(sbi.cs_number, sbi.cs_phone_number) AS cs_phone_number,
    sbi.cs_phone_number AS alim_talk_phone_number,
    sbi.registration_number,
    sbi.sale_report_number,
    sbi.representative,
    sbi.cs_email
FROM seller s
INNER JOIN seller_business_info sbi ON sbi.seller_id = s.seller_id
WHERE s.seller_id = :sellerId
LIMIT 1
```

## 마이그레이션 주의사항

### sellerName 필드의 실제 소스

응답의 `sellerName`은 `seller.seller_name` 컬럼이 아닌 `seller_business_info.company_name`을 사용한다. `seller` 테이블에 `seller_name` 컬럼이 별도로 존재하지만 이 API에서는 사용하지 않는다. 마이그레이션 시 신규 시스템에서 어떤 필드를 sellerName으로 매핑할지 결정이 필요하다.

### address 필드 조합 방식

3개 컬럼을 단순 공백으로 이어붙인다:
```
{business_address_line1} {business_address_line2} {business_address_zip_code}
```
신규 시스템에서 주소 필드를 분리된 구조로 제공할지, 동일하게 concat할지 결정 필요.

### csPhoneNumber vs alimTalkPhoneNumber 중복

shadow traffic 결과에서 두 필드가 동일한 값(`"01036817687"`)을 가지는 이유:
- cs_number가 null인 경우 COALESCE에 의해 두 필드 모두 cs_phone_number 값을 갖는다.
- cs_number가 값을 가지는 셀러는 csPhoneNumber와 alimTalkPhoneNumber가 다를 수 있다.

### Redis TTL 미설정

`saveSellerInRedis()` 호출 시 TTL이 설정되지 않아 캐시가 영구 저장된다. 셀러 정보 수정 후 캐시 수동 삭제가 필요한 구조다.
